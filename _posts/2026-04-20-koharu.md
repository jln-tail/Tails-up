Recently, I got really into *Ura Baito* (裏バイト：逃亡禁止) and started looking for a way to read the manga more comfortably and share translated screenshot snippets with friends. That led me to [Koharu](https://github.com/mayocream/koharu), a Tauri + Rust + CUDA project.

When I tried to build it on Windows, I ran into two major issues, which I’m documenting here.

## The first error: it looked like “CUDA 13.1 does not support VS2022,” but that was not the real problem

The first error looked like this:

```text
C:\Program Files\NVIDIA GPU Computing Toolkit\CUDA\v13.1\include\crt\host_config.h(164): fatal error C1189:
#error: -- unsupported Microsoft Visual Studio version! Only the versions between 2019 and 2022 (inclusive) are supported!
```

Since I originally had CUDA Toolkit 11.8 installed, my first instinct was to upgrade CUDA. After checking that my `cl.exe` version was `19.44.35222` from Visual Studio 2022, I upgraded CUDA Toolkit to 13.1, assuming that would be enough.

However, the error still persisted.

At that point, I was confused, because the versions appeared to be compatible. Then I remembered that I had also installed VS2026 earlier. Suspecting that it might be interfering with the build, I tried running `bun dev` from the **x64 Native Tools Command Prompt for VS 2022**. I assumed this would ensure that the build used the VS2022 toolchain. It still failed.

I then redirected the build output to a log file and started reading it more carefully. There, I found this line:
```text
cargo:rerun-if-env-changed=NVCC_CCBIN
```
This suggested that I could explicitly tell `nvcc` which host compiler to use. So I ran:
```cmd
set "NVCC_CCBIN=C:\Program Files\Microsoft Visual Studio\2022\Community\VC\Tools\MSVC\14.44.35207\bin\Hostx64\x64\cl.exe"
```
This way, `nvcc` no longer had to rely on whatever compiler happened to be resolved through `PATH`; it would use the exact `cl.exe` I specified.

Later logs confirmed that this really did take effect. The `nvcc` command line explicitly contained `-ccbin`, and it pointed to the VS2022 `cl.exe`.

## The second error: missing libclang

After addressing the first issue, I ran into another error:
```text
Unable to find libclang: "couldn't find any valid shared libraries matching: ['clang.dll', 'libclang.dll'], set the `LIBCLANG_PATH` environment variable..."
```
This error came from the `koharu-llm` build script, not from `candle-kernels`.

This appeared to be a separate issue. I either did not have LLVM installed properly, or `libclang` was not discoverable from the environment. I then installed LLVM:
```cmd
winget install LLVM.LLVM
```
and set:
```cmd
set "LIBCLANG_PATH=C:\Program Files\LLVM\bin"
```
After that, the build progressed without further issues.

## Short conclusion

The main lessons I took away are fairly simple:

1. The misleading part was that `cl` looked correct in my terminal, but that did not guarantee that `nvcc` would pick the same compiler internally.
2. Launching a build from a specific command prompt does not necessarily guarantee that every tool in the build chain will use the compiler I expect.
3. When multiple Visual Studio versions coexist, explicitly setting `NVCC_CCBIN` is safer than relying on implicit tool discovery.
4. Saving the full output to a log file is extremely helpful for searching and debugging later:
   ```cmd
   bun dev > .\build-full.log 2>&1
   ```
